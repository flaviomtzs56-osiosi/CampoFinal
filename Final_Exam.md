# 2. GIT, VERSIONAMIENTO Y TRABAJO PROFESIONAL

### 4. Explica con tus palabras el flujo básico de trabajo en Git (commit, push, pull, branch, merge).

El flujo básico es así: primero haces cambios en tus archivos, los agregas con `git add`, y los guardas local con `git commit`. Luego creas una rama nueva con `git branch` o `checkout -b` para no joder la principal. Cuando terminas, haces `git push` para subir tus cambios al repo remoto. Para traer lo de los demás usas `git pull` que baja y mergea automáticamente, y al final `git merge` para unir tu rama con la main cuando todo está listo.

### 5. ¿Por qué es importante usar control de versiones en proyectos reales?

Porque en proyectos reales siempre hay varios trabajando al mismo tiempo, y sin control de versiones te pierdes quién cambió qué o cómo volver si algo sale mal. Te salva de bugs, permite colaborar sin pisar código ajeno y mantiene un historial claro de todo.

### 6. ¿Cómo aplicarías GitHub en un proyecto profesional o universitario en equipo?

En un proyecto en equipo subo todo a un repo de GitHub, creo branches para cada feature, hago pull requests para que todos revisen antes de mergear a main, y uso issues para trackear tareas. Así todos vemos el progreso y evitamos conflictos.
